# Dev Clipboard 日本語ハンドオフ

## 暫定決定

プロダクト名:

**Dev Clipboard**

日本語キャッチコピー:

**“貼る前に理解する” 開発用クリップボード**

英語キャッチコピー:

**Understand before you paste.**

## 背景

このプロダクトは、AIを使った開発中に感じる実務上の不安から出発している。

フロントエンド開発も行うアートディレクター、デザイナー、プロダクトビルダー、非バックエンド寄りの開発者は、ドキュメントやAIの回答を見ながらターミナルコマンド、ファイルパス、パッケージマネージャー、スラッシュコマンド、設定ファイルなどをコピー&ペーストする場面が多い。

Paste、Maccy、PastePal、Alfred、Raycastなどの汎用クリップボードマネージャーは、コピー履歴を保存して再利用する用途では非常に便利。一方で、AI開発時の「このコマンドを本当に貼って大丈夫か」「これは何をするものか」「どのパスを置き換えるべきか」といった開発特有の不安には、まだ深く最適化されていない。

Dev Clipboardの機会は、汎用クリップボードではなく、AI時代の開発作業に特化した「貼る前の理解」を支援する点にある。

## 初期課題

AIを使った開発では、コピーする情報の量と種類が増える。

代表例:

- ドキュメントやAI回答からコピーしたShellコマンド。
- `rm`, `mv`, `chmod`, `sudo`, `git reset`, `docker system prune` などの危険になり得るコマンド。
- 現在のプロジェクトやカレントディレクトリに依存するファイルパス。
- AIツール、エディタ、CLIで使うスラッシュコマンド。
- 環境変数、JSON、Markdown、SQL、パッケージマネージャーのコマンド、設定断片。
- ターミナルやブラウザからコピーしたエラーログ。

問題の本質は、コピー履歴を忘れないことだけではない。

より深い課題は、コピーした内容の意味、安全性、使い方を理解しないまま、ターミナルやエディタに貼ってしまうことにある。

## 目的

Dev Clipboardは、開発中にコピーした情報を、貼る前に理解できる状態にする。

目指すこと:

- コピーしたコマンド、スニペット、パス、プロンプト、エラー、ドキュメント断片を保存する。
- 内容を自動分類する。
- コマンドやスニペットが何をするものか説明する。
- リッチテキスト、プレーンテキスト、Markdown、コード、URL、Shellコマンド、JSON、SQL、`.env`、スラッシュコマンドなどを一目で判別できるようにする。
- 保存済みのコピー情報に、簡単な説明や注意書きを付けられるようにする。
- 変数付きスニペットと、貼る直前の入力補助を提供する。
- ターミナルやエディタへの危険なペーストを減らす。

## 2026-06-27 時点の重要な整理

本日の議論で、Dev Clipboardの中心価値は「開発用のコピー履歴」ではなく、より明確に以下へ寄った。

**コピーした情報に“意味”を付けて、安心して再利用できる保管庫。**

Pasteが評価されているのは、ショートカットでいつでも呼び出せること、コピーした情報が視覚的なカードとして分かりやすいこと、ドラッグ&ドロップで直感的に別アプリへ持っていけること、検索やカテゴリーで探しやすいこと、保存期間を長く保てることにある。

Dev Clipboardは、その気持ちよさを参考にしながら、主役を「履歴」ではなく **メモ付きクリップ** に置く。

特にコードやコマンドは、本文だけでは再利用時に不安が残る。そこで、各クリップに以下のような補足情報を持たせる。

- これは何か。
- 何をするか。
- いつ使うか。
- どこを変更するか。
- 実行前に何を確認するか。
- 危険な点は何か。
- どのプロジェクト/文脈で使ったか。

この設計により、ユーザーは「過去にコピーしたものを雑に再利用する」のではなく、「意味と注意点を確認してから安全に再利用する」体験を得られる。

キャッチコピーである **“貼る前に理解する”** と最も噛み合う中核機能は、このメモ付き保管庫である。

## 2026-06-29 時点の重要な整理

スパイク実装を通じて、メモ付きクリップの基本構造は以下の3つに整理した。

- `Description`: これは何か。何をするものか。
- `When to use`: いつ使うか。どの状況で再利用するか。
- `Before`: 貼る前/実行前に確認すること。

UI上では、`Before` は安全確認として優先度が高いため、Terminalカードでは常時見える情報として扱う。`Description` と `When to use` は、カード上で短く見せるか、必要に応じて詳細表示へ下げる。どちらもBeforeと同じようにユーザーが後から編集でき、検索対象にも含める。

検索は単に本文だけを探すものではなく、ユーザー自身が付けたメモ、危険度、Before、Dev向けメタデータまで含めて探せることが価値になる。たとえば本文が `rm -rf dist` でも、`再帰的に削除` や `ビルド成果物` といった言葉で見つけられる体験を重視する。

また、Dev Clipboard自身のUIには多くの説明テキストが表示されるため、アプリ内でテキストをコピーしたものが履歴に混ざる問題がある。MVPでは、Dev Clipboardがフォーカスされている間のコピーは保存しない方針にした。ユーザーがメモ編集中に別アプリからコピーした内容をDev Clipboardへ自動保存できないケースはあるが、履歴汚染を防ぐほうを優先する。

## 2026-06-30 時点のMVP境界

MVPでは、機能を広げすぎず、以下の価値に集中する。

**貼る前に、コピー内容の意味・使いどころ・注意点を理解できること。**

MVPに入れるもの:

- Chat / Editor / Terminal / All Vaults
- クリップボードのプレーンテキスト保存
- ローカルSQLite保存
- クリップ本文、メモ、Risk、Beforeを含む検索

- Compact / Normal / Largeのカード表示
- 検索ヒット元表示

- Local only、Privacy、Storage、Safety、Display、Shortcuts、Helpの設定画面

MVPに入れないもの:

- チーム共有
- ログイン必須体験

将来的なフェーズ:

1. Phase 0: 技術スパイク。Tauri、SQLite、検索、メモ編集、カードUI検証。
2. Phase 1: MVP。毎日使えるローカルファーストのDev Clipboard。
3. Phase 2: Daily Driver化。アプリアイコン、検索ソート、カード削除、Shiki、Markdown表示、ショートカット編集。
4. Phase 3: Intelligence。AIメモ生成、日本語検索強化、セマンティック検索、リッチデータ対応。
5. Phase 4: Ecosystem。クラウド同期、MCP、チーム共有、課金/ライセンス。

## コンセプト

**Dev Clipboardは、AI開発のための作業記憶であり、貼る前の安全確認レイヤーである。**

一般的なクリップボード履歴アプリのように「過去にコピーしたものを探す」だけではなく、開発文脈に合わせて「それが何で、どう使うべきか」を示す。

Pasteのような汎用クリップボードマネージャーと正面から戦うのではなく、AI支援開発、コマンド、スニペット、ターミナル操作に特化することで差別化する。

## ポジショニング

Dev Clipboardは以下の中間に位置する。

- クリップボード履歴アプリ
- スニペットマネージャー
- AIコーディングツール
- ターミナル安全確認ツール
- 開発ドキュメント支援ツール

ポジショニング案:

**貼る前に、文脈・意味・危険度を理解できる開発用クリップボード。**

追加のポジショニング案:

**コピー履歴をただ残すのではなく、あとで安全に使えるように、意味と文脈を添えて保管する開発用クリップボード。**

## 想定ユーザー

主なユーザー:

- AIツールを使うフロントエンド開発者。
- コーディングも行うデザイナー、アートディレクター。
- 個人開発者、プロダクトビルダー。
- バックエンドやターミナル操作に強い自信がない開発者。
- Cursor、Codex、Claude、ChatGPT、VS Code、iTerm、Terminal、Warp、Raycastなどを行き来するユーザー。

副次的なユーザー:

- コマンドやスニペットを構造化して保存したい経験豊富な開発者。
- チーム内で安全なコマンドやオンボーディング用スニペットを共有したい人。
- ドキュメント、AIチャット、ターミナル、エディタを頻繁に行き来する人。

## 初期アイディア

### 1. 開発文脈に強いクリップボード履歴

コピー履歴に開発向けのメタデータを付けて保存する。

分類候補:

- プレーンテキスト
- リッチテキスト
- Markdown
- Shellコマンド
- Gitコマンド
- パッケージマネージャーのコマンド
- URL
- ファイルパス
- JSON
- SQL
- 環境変数
- エラーログ
- コードスニペット
- スラッシュコマンド
- AIプロンプト

### 2. コマンド理解

コピーした内容がコマンドらしい場合、短い説明を表示する。

例:

```sh
rm -rf dist
```

表示例:

`dist` ディレクトリを再帰的かつ強制的に削除する。破壊的な操作であり、現在のカレントディレクトリに依存する。

表示したい情報:

- コマンド名
- 引数
- フラグ
- 対象パス
- 危険度
- 実行前に確認すべきこと

### 3. 危険度判定

注意が必要なコマンドを貼る前に示す。

危険度カテゴリ案:

- Safe
- Caution
- Destructive
- Requires confirmation
- Unknown or suspicious

警告対象の例:

- `rm`, 特に `rm -rf`
- 上書きや大きな移動を伴う `mv`
- `chmod`, `chown`
- `sudo`
- `curl | sh`
- `git reset --hard`
- `git clean`
- `docker system prune`
- `npm publish`
- 認証情報、トークン、シークレットを含む可能性がある文字列

### 4. ペースト前プレビュー

貼る直前に、コンパクトなプレビューを表示する。

表示内容:

- コンテンツ種別
- フォーマット状態
- 短い説明
- 危険度
- 貼り先に応じたオプション

ペーストモード候補:

- そのまま貼る
- プレーンテキストで貼る
- Markdownコードブロックとして貼る
- 自動ペーストせずコピーだけする
- プレースホルダーを入力してから貼る

### 5. リッチテキスト/プレーンテキストの可視化

コピーした内容がどの形式かを一目で分かるようにする。

表示候補:

- Plain Text
- Rich Text
- Markdown
- Code
- URL
- Image
- File

エディタ、Issue管理、AIチャット、ターミナルでは、意図しない書式が邪魔になることがあるため、形式の可視化は重要。

### 6. メモと説明

保存済みクリップに短い説明を追加できるようにする。

例:

- このコマンドが何をするか。
- いつ使うか。
- どのプロジェクト用か。
- 実行前に何を確認するか。
- どのAI/ツール/ドキュメントから得たものか。

「なんとなく覚えているが、自信がないコマンド」を再利用するときに効く。

現時点では、この機能を最重要機能とする。

メモは単なる自由入力欄だけでなく、以下のような型を持たせるとよい。

- **Description**: これは何か。
- **When to use**: いつ使うか。
- **Before paste**: 貼る前に確認すべきこと。
- **Variables**: 置き換える値。
- **Risk**: 注意点、危険な点。
- **Source**: どこからコピーしたか。
- **Last used context**: 最後に使った文脈。

カード一覧では本文とメモ要約を見せ、詳細ビューでこれらを編集できるようにする。

自動メモは段階的に考える。

AIなしで可能:

- コマンド名から基本説明を付ける。
- `rm`, `sudo`, `chmod`, `git reset` などを危険判定する。
- URL、パス、カラー、JSON、Markdown、コードを分類する。
- `{{variable}}` や `$ENV` のような変数を検出する。
- コピー元アプリ、日時、使用回数、最終使用日を保存する。

AIがあると強い:

- コードやコマンドの自然言語説明。
- いつ使うかの説明。
- 危険箇所の要約。
- スニペット名の提案。
- タグの自動付与。

MVPでは、**手動メモ + ルールベース自動メモ** を優先し、AIメモ生成は後続フェーズで検討する。

### 7. 変数付きスニペット

再利用するコマンドやスニペットに変数を持たせる。

例:

```sh
cd {{project_path}}
git checkout -b {{branch_name}}
pnpm add {{package_name}}
scp {{local_file}} {{server}}:{{remote_path}}
```

選択時に小さな入力フォームやツールチップを出し、必要な値を埋めてから貼れるようにする。

### 8. AI/MCP連携

PasteもMCP連携に対応しているが、Dev Clipboardでは開発文脈に特化したMCP連携にする。

利用例:

- AIが許可された最近の開発クリップを参照する。
- AIが選択中のコマンドを説明する。
- AIが危険なコマンドをより安全な形に書き換える。
- AIがコピーしたコマンドから再利用スニペットを作る。
- AIがエラーログを要約する。
- AIが意図から保存済みスニペットを検索する。

重要な方針:

AIに全クリップボード履歴を無条件で渡さない。アクセスは明示的かつスコープ化する。

2026-06-27時点の判断:

MCPは重要な将来機能ではあるが、初期MVPには入れない。

理由:

- MCPなしでも「メモ付き保管庫」「貼る前の理解」「危険コマンド警告」で価値を出せる。
- まずは気軽に使ってもらうハードルを下げたい。
- AI連携より前に、日常的な呼び出し、一覧性、検索、メモ付け、安心感を磨く必要がある。
- セキュリティ説明や権限制御が未成熟なままMCPを入れると、不安が先に立つ可能性がある。

### 9. 貼り先に応じた動作

貼り先のアプリによって、表示や補助を変える。

例:

- Terminal: コマンド説明、危険度、プレーンテキスト貼り付けを優先。
- VS Code/Cursor: コードスニペット、ファイルパス、プロンプト、スラッシュコマンドを優先。
- ブラウザ/AIチャット: Markdown整形、コードフェンス、プロンプトスニペットを優先。
- GitHub/Linear/Notion: Issueやドキュメント向けの整形を優先。

### 10. 初期保管庫

初期設定では、以下の3つの保管庫を用意する。

1. **Chat**
   - ChatGPT、Claude、Codex、Cursor Chatなどに貼る情報。
   - プロンプト、回答、エラー文、調査メモ、URL、Markdownを保存。
   - Markdown整形、コードフェンス化、引用化が重要。

2. **Editor**
   - VS Code、Cursor Editor、Xcodeなどに貼る情報。
   - コード断片、コンポーネント名、ファイルパス、CSS、カラーコード、JSON、設定ファイルを保存。
   - カラー変換、インデント保持、リッチ/プレーン切替が重要。

3. **Terminal CLI**
   - Shell、Git、npm/pnpm、Docker、brew、curl、chmod、rmなど。
   - 危険度判定、説明、確認、変数入力、実行前チェックが重要。
   - Dev Clipboardの差別化が最も出やすい保管庫。

ユーザーは任意で保管庫/カテゴリーを追加できる。

### 11. 複数選択と結合ペースト

Terminal CLI保管庫では、複数のコマンドを選んで1つの複数行コマンドとして結合できるとよい。

想定操作:

- Shift または Command を押しながら複数カードを選択。
- 下部に選択バーを表示。
- `Copy combined` / `Paste combined` / `Preview` / `Clear selection` を表示。

例:

```sh
cd {{project_path}}
pnpm install
```

重要な判断:

**順番に自動ペーストする機能は初期では入れない。**

理由:

- 便利さはあるが、ユーザーが入力先やタイミングを誤る事故が起きやすい。
- 特にターミナルでは、自動連続ペーストは安全性よりリスクが上回る可能性がある。
- Dev Clipboardでは自動実行感よりも、プレビューして理解してから貼る体験を優先する。

複数行に結合した場合は、個別コマンドではなく、結合後の全体に対して危険度判定を行う。

### 12. カラーコード変換

デザイナー/フロントエンド開発者向けに、カラーコードの事前変換は重要。

例:

- `rgb(255, 148, 0)` -> `#FF9400`
- `#FF9400` -> `rgb(255, 148, 0)`
- `hsl(...)` -> HEX/RGB

Editor保管庫では、カラーコードをカード上でスウォッチ表示し、ペースト前に形式を選べるようにする。

### 13. エディタテーマ風コードプレビュー

Editor保管庫では、コードクリップを普段使っているエディタに近い見た目で表示できるとよい。

例:

- Cursor/VS Codeで使っているコードカラーテーマに近い配色で表示する。
- HTML、CSS、TypeScript、JSON、Shellなどを言語ごとにシンタックスハイライトする。
- 背景色、文字色、属性名、文字列、タグ名、コメントなどを見慣れた色にする。
- Editor保管庫はCursor風、Terminal CLI保管庫はターミナル風、Chat保管庫はMarkdown風のように保管庫ごとに表示テーマを変える。

この機能は単なる装飾ではなく、貼る前の理解と安心感に効く。

ユーザーが普段見ているCursor/VS Codeに近い見た目でコードが表示されると、以下の効果がある。

- コードかどうか即判断できる。
- 言語構造が見えやすい。
- ペースト前の違和感が減る。
- 「これを貼って大丈夫そうか」の確認が速くなる。
- Editor保管庫の体験が実務エディタとつながる。

技術的には実現可能。

実装候補:

- VS Code/CursorのテーマJSONを読み込む。
- TextMate grammarでコードをトークン化する。
- ShikiやMonaco Editorを使ってテーマ付きでレンダリングする。
- コードクリップのカード/詳細ビューにそのテーマを適用する。

現実的な実装ステップ:

1. コードを自動判定する。
2. 言語を推定する。
3. まずは標準ダークテーマでシンタックスハイライト表示する。
4. ユーザーがテーマを選べるようにする。
5. VS Code/CursorテーマJSONをインポートできるようにする。
6. 将来的にCursor/VS Code設定から自動取得する。

注意点:

Cursor/VS Codeの見た目を100%完全再現するのは難しい。

理由:

- カラーテーマ。
- TextMate grammar。
- semantic highlighting。
- フォント。
- ligature。
- 行間。
- インデントガイド。
- bracket pair colorization。
- 選択色。
- エラー/警告装飾。
- 拡張機能の独自表示。

MVPでは完全再現ではなく、**Cursorで見ている感覚に近いコードカード** を目標にする。

2026-06-27の簡易検証:

`experiments/code-theme-preview/` にShikiベースのHTMLプロトタイプを作成し、添付画像のような黒背景、水色タグ、黄色属性、ピンク文字列のコードカード表示が可能であることを確認した。

検証結果:

- VS Code互換のテーマオブジェクトで、Cursor風の配色に近づけられる。
- コードカードとメモパネルを並べるUIは視認性が高い。
- 長い行はカード幅、フォントサイズ、横スクロール設計の調整が必要。
- MVPでは「完全なエディタ再現」ではなく「見慣れた配色で構造が読みやすいプレビュー」で十分価値がある。

機能名候補:

- Editor Theme Preview
- Code Theme Sync
- Cursor-style Preview
- Match Editor Theme
- Code Appearance

### 14. 画像素材の整形

マルチモーダル時代には、画像も単なる履歴ではなく、素材として再利用しやすく整える価値がある。

例:

- アプリのアイコンを白い余白込みでキャプチャーした場合、外側の無駄な余白を自動クロップする。
- 単色背景や透明背景を検出して、対象物だけの画像に整える。
- 余白量を指定して再保存する。

初期はAIなしでも以下は可能。

- 外周の背景色検出。
- 白/透明/単色背景の自動トリミング。
- 指定余白を残したクロップ。
- PNGとして再保存。

高度化する場合:

- 被写体検出。
- 背景除去。
- 影の保持。
- アイコン/ロゴ/スクリーンショットの種類判定。
- 1x/2x/3x書き出し。

これはMVP外だが、デザイナー視点の差別化機能として有望。

### 15. 一覧性と表示密度

Pasteの評価から、コピーした情報の一覧性は重要。

必要な表示モード:

- **Quick overlay**: ショートカットで呼び出し、すぐ貼る。
- **Fullscreen**: 探す、整理する、保管庫を編集する。
- **Compact cards**: テキスト履歴を多く見る。
- **Normal cards**: 本文とメモ要約を見る。
- **Large cards**: 画像、コード、リッチプレビューを大きく見る。

「呼び出して貼るUI」と「整理するUI」は分ける。

Dev Clipboardでは、カードの主情報を「本文」だけでなく「本文 + メモ要約 + 種別 + 危険度」として設計する。

### 16. Context Size / Storage Weight

AIに渡す情報量や、保管庫を圧迫するデータサイズも見える化する。

理由:

- AIに渡す文字量はトークン消費に関係する。
- 長いログ、Markdown、JSON、コードはAIに渡す前に重さを判断したい。
- 画像、Illustrator由来のコピー、リッチテキスト、スクリーンショットなどはストレージ/メモリを圧迫しやすい。
- Dev Clipboardの「貼る前に理解する」は、意味だけでなく、情報量・コスト・重さも含められる。

MVPでは機能実装よりも、まず画面要素として表示する。

表示例:

- `~240 tokens`
- `~3.2k tokens`
- `86 lines`
- `3 code blocks`
- `4.8 MB`
- `Large`
- `AI cost: High`

カード種別ごとの例:

- Chat: `Markdown · ~1.8k tokens · 3 code blocks`
- Editor: `TSX · 86 lines · ~1.2k tokens`
- Terminal CLI: `Shell · 2 lines · Low context`
- Image: `PNG · 2400 x 1600 · 4.8 MB`

重要な判断:

- MVPでは正確なモデル別トークン計算は不要。
- `~` を付けた推定表示、またはLow/Medium/Highでよい。
- データサイズもまずは見た目としてカード/詳細ビューに入れる。
- サイズ順ソート、削除、圧縮、要約は後続で検証する。

将来機能:

- サイズ順ソート。
- トークン数順ソート。
- 重いクリップ一覧。
- 30日未使用かつ大きいクリップの削除候補。
- 画像はサムネイルだけ保存。
- AIに渡す前に要約/軽量化。

### 17. セキュリティとプライバシー

クリップボードアプリは、ユーザーが意識していない秘密情報も拾う可能性がある。

初期から設計に入れるべき項目:

- ローカル保存を基本にする。
- クラウド同期は初期OFF、または初期MVPでは未実装。
- パスワードマネージャー、認証アプリ、銀行系アプリなどは自動除外。
- `password`, `token`, `secret`, `api_key`, `BEGIN PRIVATE KEY` などを検知。
- 検知したものは保存しない、マスクする、短時間で削除するなどを選べる。
- 保管庫ごとに保存期間を設定できる。
- クリップごとに「永久保存」「30日で削除」「使用されなければ削除」を選べる。
- Terminal CLI保管庫は危険コマンドを短期保存にできる。
- 初回起動時に、何を保存し、何を保存しないかを明確に説明する。

設定画面には **Privacy & Retention** を独立した大項目として置く。

### 18. 保存期間と自動削除

開発者のコピー履歴はすぐに汚れるため、自動整理は重要。

ルール案:

- 保管庫ごとに保持期間を設定。
- デフォルト例:
  - Chat: 30日
  - Editor: 90日
  - Terminal CLI: 30日
- 固定したものは永久保存。
- スニペット化したものは永久保存。
- 30日未使用なら自動削除。
- 危険コマンドは短期保存、または保存しない設定も可能。
- シークレットらしき文字列は自動除外/即時削除。

### 19. クラウド同期

クラウド同期はセキュリティ上のハードルが高いが、ニーズは強い。

特に、メモ付きクリップ、保存済みスニペット、保管庫構成は複数端末で使いたくなる可能性が高い。

初期方針:

- クリップボード履歴全体の同期はデフォルトOFF。
- 保存済みクリップは同期対象にできる。
- Secret判定されたものは同期禁止。
- Terminal CLI保管庫は同期OFFをデフォルト。
- Chat / Editor保管庫はユーザー選択で同期可能。
- 個別アイテムごとに「同期する/しない」を選べる。

Obsidian Syncのように、ローカルファーストで有料同期を後から足す設計はあり。

ただし、暗号化、競合解決、削除同期、端末間の整合性が必要になるため、MVPでは後回しが現実的。

## MVP候補

MVPは **Minimum Viable Product** の略で、「最小限成立するプロダクト」を意味する。

ただし、機能が少ない雑な試作品ではない。

Dev ClipboardにおけるMVPは、最重要価値である **メモ付きクリップによって、貼る前に理解して安全に再利用できること** を検証できる最小構成とする。

初期バージョンで狙う範囲:

1. Chat / Editor / Terminal CLI の3保管庫。
2. クリップを保存できる。
3. 各クリップにメモを付けられる。
4. カード一覧で本文 + メモ要約 + 種別が見える。
5. 詳細ビューで説明/使う場面/注意点を編集できる。
6. コマンド、コード、URL、カラー、画像などを自動分類する。
7. Terminal CLIでは危険コマンドに説明と警告を出す。
8. 推定トークン数、行数、データサイズなどを表示できる。
9. ショートカットで呼び出せる。
10. 検索できる。
11. 複数選択して改行結合プレビューができる。
12. 保存期間/自動削除を設定できる。
13. シークレット検知/保存除外ができる。
14. プレーンテキスト/そのまま貼るを選べる。

MVPに入れないもの:

- MCP連携。
- AIによる高度な説明生成。
- クラウド同期。
- 画像の高度な自動クロップ。
- 自動順次ペースト。
- 高度な貼り先別制御。

これらは魅力的だが、最初に検証すべき価値ではない。

まずは **Pasteの気持ちよさ + 開発用3保管庫 + メモ付きクリップ + Terminal CLI安全性** を成立させる。

## デザイン方針

UIは、実務で使いやすく、精密で、落ち着いた印象がよい。

開発者向けツールであるため、過度にマーケティング的な見せ方よりも、情報の読みやすさ、信頼感、スピードを重視する。

UIアイディア:

- シンタックスハイライト付きのコンパクトなコマンドカード。
- コンテンツ種別を示す明確なラベル。
- 色だけに頼らない危険度表示。
- パス、フラグ、URL、変数のハイライト。
- 保存済みクリップには1行メモ要約を表示。
- ツールチップのような説明。
- 変数入力用の小さなフォーム。
- キーボードで素早く選べるペーストモード。
- 単なるコピー履歴と、意図的に保存したスニペットの視覚的な区別。
- Compact / Normal / Large のカードサイズ切り替え。
- Fullscreenで一覧・整理できる画面。
- Quick overlayで即ペーストできる画面。
- Editor保管庫のコードカードにエディタテーマ風プレビューを適用。
- コードテーマを保管庫ごと、または言語ごとに切り替え。

技術的な内容を隠しすぎず、怖さを減らすUIが重要。

カードの基本構造案:

- 上部: 種別バッジ、保管庫、危険度、最終使用日。
- 中央: コピー本文のプレビュー。
- 下部: メモ要約、変数、実行前チェック。
- 詳細: Description / When to use / Before paste / Variables / Risk / Source。

Editor保管庫のコードカードは、ユーザーが普段使うCursor/VS Codeの視覚体験に近づける。

これにより「コードを見慣れた状態で確認してから貼る」ことができ、ペーストへの抵抗を減らす。

## 2026-06-28 技術スパイク結果

`apps/dev-clipboard-spike` に、Tauri 2 + React + TypeScript の技術スパイクを作成した。

確認できたこと:

- macOSの実クリップボードを監視できる。
- コピーしたテキストがDev Clipboard Spikeにカードとして反映される。
- `rm -rf dist` のようなTerminalコマンドを簡易Risk付きで表示できる。
- アプリ側の `Copy` / `Review` からシステムクリップボードへ戻せる。
- ユーザーが別アプリでCommand+Vすれば貼れる。
- Auto PasteやAccessibility権限なしで、MVPの最小ループが成立する。
- SQLite保存が動作する。
- Spikeを閉じて再起動しても、保存済みカードが復元される。
- SQLite FTS検索が動作する。
- `rm -rf dist` を `再帰的に削除`, `再起的に削除`, `再帰削除`, `再起削除`, `dist削除` で検索できる。
- 検索結果に `Matched in Body` / `Matched in Dev metadata` とヒット理由を表示できる。
- 検索語そのものが履歴に入る問題に対して、検索欄と同じ文字列は保存しないガードを追加した。
- 0件時は通常の空状態ではなく、検索0件として表示する。
- インラインBeforeメモ編集が動作する。
- 編集したBeforeメモはSQLiteに保存され、FTSへ再インデックスされる。
- `ls dist` で `rm -rf dist` が `Matched in Before` としてヒットすることを確認した。
- Dev Clipboardがフォーカス中のclipboard changeを保存しない内部コピーガードが動作する。
- `Internal copy guard on` / `External capture on` の状態バッジを確認した。

技術的に分かったこと:

- Tauriの `@tauri-apps/plugin-clipboard-manager` でテキストのread/writeは検証できた。
- Tauriの `@tauri-apps/plugin-sql` でSQLite保存は可能。
- `sql:default` はload/select中心で、INSERT/UPDATE/DELETEには `sql:allow-execute` が必要。
- DBは `~/Library/Application Support/com.department.devclipboard/dev-clipboard-spike.db` に作成された。
- SQLite FTSは導入済みだが、日本語の分かち書きは弱いため、Dev modeでは検索用メタデータを厚くする方針が合う。
- 表記ゆれは、Dev modeの検索用メタデータで吸収できる。例: `再帰的に削除` / `再起的に削除`。
- Dev Clipboard自身の検索欄やメモ編集欄からコピーした文字列が履歴に入るとノイズになるため、アプリ内コピーを保存しないガードを追加した。

検索方針:

- 検索UIは共通化する。
- 裏側の検索エンジンは `Dev mode` / `Plain mode` で切り替える。
- Dev modeはSQLite FTS + Risk / Before / Description / Tagsなどの検索用メタデータを使う。
- Plain modeは将来的にN-gramまたは日本語tokenizerを検討する。
- 最初から重い日本語検索エンジンを入れず、拡張口を残す。

## 2026-07-02 時点の重要な整理

本日は、Tauri/React Spike上でカードUI、リッチプレビュー、色変換、スライドインパネルの使い心地を詰めた。

今日の大きな前進:

- Dev Clipboardは、通常ウィンドウではなく、左からせり出す常駐パネルとして成立する見込みが強くなった。
- `Command + Option + V` で表示/退避し、フォーカスアウトで自然に閉じる動きが確認できた。
- パネル外を完全透明にし、カード本体は不透明にする方向で、作業中アプリの上に浮く見え方を検証した。
- シャドウ、角丸、リサイズ、レスポンシブは見た目への影響が大きいため、今後も優先して検証する。
- カードの主役は、本文だけでなく「コピー元アプリ」「Safety」「保管庫」「Type/Media」「メモ」で判断できる構造へ進んだ。
- URL、Color、Image、SVG、Illustrator/Figma系データ、Audio、Video、File/PDFの表示ルールを整理し始めた。

今日決めたカード方針:

- タグの優先順位は `Safety -> Vault -> Type/Media -> size/lines/tokens/extra`。
- `Safe`, `Review`, `Risk` はアプリの性格を決める重要タグとして、塗りの強いピルで扱う。
- `Review` や `Risk` の理由文言は、タグ名そのものではなく、Details、Risk breakdown、検索理由へ逃がす。
- コピー元アプリは、タグ列ではなくタイトル左のアプリアイコンとして出す。アプリ名は表示しない。
- タイトルと編集アイコンは文末に置き、ユーザーが自分で管理しやすいようにタイトル編集を可能にする。
- `Description`, `When to use`, `Before` は空でも表示し、すぐ編集できる導線として検証する。自動メモ生成は次段階。
- 右上の主ボタンは、基本 `Copy`。注意確認が必要なものは `Review` または `Risk` として出す。
- コピー完了時は、緑系背景 + チェックアイコン + `Copied` で明確にフィードバックする。

Colorの方針:

- Colorは、Pasteとの差別化につながる重要な表示タイプ。
- `#hex` だけでなく、`rgb(100,100,100)` と `rgb(100, 100, 100)` の両方、前後スペース、RGBA、8桁HEX、HSL/HSLAも扱える設計にする。
- 検出や変換ではtrim/正規化するが、元のコピー本文は保持する。
- ペースト/通常Copyは、基本的に元の文字列を尊重する。
- Colorカードでは、HEX/RGB/HSL/RGBAなどのボタンで表示形式を切り替え、その表示中の値を右上のCopyでコピーする。
- 自動タイトルはColorに限り表示形式と連動してよい。ただしユーザーが独自タイトルを付けた場合は固定タイトルとして扱う。
- Colorは形式変換だけでなく、色そのものを変更できるカラーピッカー機能も将来検討する。RGB -> HEXなどの表記変換とは役割が違うため、変換ボタンとは別のUIとして設計する。

リッチプレビュー方針:

- URLはプレビュー内のURL自体をリンクとして扱い、ホバーで下線を出す。主ボタンにOpenは混ぜない。
- Audio/Videoは中央の大きい再生ボタンではなく、メディアプレビュー右下の小さい再生アイコンとして扱う。
- Audioは `AudioLines` 系のアイコンを採用し、Video/Image/SVG/Fileと同じ視覚重量に寄せる。
- Figma/Illustratorなどのデザインデータは、source app、file type、local path、sizeなどの属性をプレビューにも出す。
- Colorのプレビュー色は、アプリのPrimary黄色と混同しない色で検証する。
- `python.py` / `javascript.js` などの開発ファイルは、コピー内容で扱いを分ける。エディタ上でファイル内容をコピーした場合は `Editor / Code` として扱い、Finder等でファイル実体やパスをコピーした場合は `Media / File` として扱う。拡張子は検索メタデータや詳細情報に保持する。

入力支援/プリセットの方針:

- Dev Clipboardは、単なる履歴アプリではなく「理解付きの入力支援」として伸ばせる。
- ただし最初から大量のプリセットを入れると押し付けになる。
- 将来的には、設定画面から任意で入れられる `Clip Packs` / `Starter Packs` として扱うのがよい。
- 例: Terminal Starter Commands、Git Commands、AI Chat Prompts、React Snippets、Design/Color Utilities。
- Packには本文だけでなく、`title`, `description`, `when to use`, `before`, `risk`, `tags`, `search metadata` を含める。
- インストール後は通常クリップと同じように編集/削除できる。

積み残し:

- macOS/WebViewのスクロールバー背景が、スクロールバー位置にマウスがあるとグレー表示される問題。CSSの透明指定だけでは解決しきれていないため、カスタムスクロール領域、ネイティブスクロールバー非表示、またはTauri/WebView側の制約確認が必要。
- Shiki由来のチャンクサイズ警告。本番前に必要言語だけを読み込む構成へ変える。
- Light modeのカラー、シャドウ、カード階層、Safety色の再検証。
- 検索フィルターのカテゴリ順、タグサイズ、吹き出し位置、選択済みタグの見せ方。
- source appアイコンの本番取得。MVPでは「コピー時点の前面アプリ」をsource appとして記録する。
- Starter Pack/Clip Packのデータ形式と設定画面内の導線。
- リッチテキストをプレーンテキストへ変換する機能。Colorのようにカード内で変換形式を切り替えるか、右クリックメニュー/詳細メニューから `Convert to plain text` を選ぶかは未決。貼り先の事故を減らす機能として、適切なタイミングでUIを詰める。
- ヘッダー検索窓の最終レイアウト。Normal/Compact/Largeで幅いっぱいに開く挙動、選択済みタグとキャレットの距離、Popoverアイコン位置は検証中。機能実装を先に進め、デザイン整理時に再調整する。

明日の進め方としては、まずLight modeか検索フィルターのどちらかを単独テーマとして扱うのがよい。両方を同時に触ると、色の問題と操作フローの問題が混ざるため、まずは検索フィルターの情報設計を固め、その後Light modeで見た目を調整する順番が堅い。
